新增三个审批回调函数：

```typescript
const handleApprove = async (taskId: string) => {
  updateHitlStatus(taskId, "approved");
  setStatus("streaming");
  try {
    await resumeChat(taskId, "approve");
    // resume 成功后不需要重新 setCurrentTaskId — 同一 task_id，EventSource 仍在连接中
  } catch (error) {
    const message =
      error instanceof Error ? error.message : "恢复执行失败";
    setError(message);
    toast.error("审批操作失败，请重试");
  }
};

const handleReject = async (taskId: string) => {
  updateHitlStatus(taskId, "rejected");
  setStatus("streaming");
  try {
    await resumeChat(taskId, "reject");
  } catch (error) {
    const message =
      error instanceof Error ? error.message : "恢复执行失败";
    setError(message);
    toast.error("审批操作失败，请重试");
  }
};

const handleFeedback = async (taskId: string, feedbackMessage: string) => {
  updateHitlStatus(taskId, "feedback");
  setStatus("streaming");
  try {
    await resumeChat(taskId, "reject", feedbackMessage);
  } catch (error) {
    const message =
      error instanceof Error ? error.message : "恢复执行失败";
    setError(message);
    toast.error("反馈提交失败，请重试");
  }
};
```

关键点（per D-07）："反馈" 走 `reject` + `message` 路径，不是单独的类型。

**3. 关于 SSE 连接恢复（per D-09）的关键理解：**

resume 后后端继续向同一 task_id 的 Redis Stream 写入事件。前端的 EventSource 仍在连接 `/api/chat/stream/{taskId}`，后端的 `read_events` 会继续 XREAD 到新事件并推送。因此：
- 不需要创建新的 EventSource
- 不需要修改 currentTaskId
- status 从 "interrupted" 改为 "streaming" 即可
- useSSE 的 token/tool/done 事件监听器继续工作
- 新内容追加到同一条 assistant 消息（因为 addHitlSegment 已在最后追加了空 text segment）

这是设计的巧妙之处：整个 resume 流程只需要调用 API + 更新 UI 状态，SSE 连接和消息追加机制自动工作。

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-08 | T (Tampering) | handleApprove/Reject taskId | mitigate | taskId 来自 store 中的 HitlSegment（由 SSE 事件写入），不来自用户输入，后端还会校验 task status |
| T-09-09 | D (Denial of Service) | resume API 错误处理 | mitigate | try/catch 包裹 resumeChat 调用，错误时 setError + toast 提示，不会阻塞 UI |
